# 05 · 微前端通信与状态同步

微前端最大的挑战不是"加载"，而是"通信"。5 个独立应用，状态怎么共享？本章讲透本项目的 4 个通信机制。

## 难点：微前端之间默认是隔离的

每个微前端是独立打包的 JS，它们互相看不到对方的 React 状态。举几个真实问题：

1. **购物车角标**：商品微前端加购后，Shell 顶部的角标怎么 +1？
2. **登录态**：auth 微前端登录后，Shell 头部怎么显示用户名？商品页怎么知道用户已登录？
3. **路由跳转**：商品页点"去结算"，怎么跳到 order 微前端的 /checkout？
4. **共享数据**：5 个应用都要用 react，怎么保证只加载一份？

本项目用 4 个机制解决。

## 机制 1：事件总线（发布订阅）

> 代码：`shared/bus.js`（shell 里也有一份 `shell/src/bus.js`，内容一致）

这是跨微前端通信的**主力方案**。原理是把一个事件总线挂到 `window` 上，所有微前端共用同一个。

```js
function createBus() {
  const listeners = {};  // { 事件名: [回调函数数组] }
  return {
    on(event, cb) {                          // 订阅
      (listeners[event] ||= []).push(cb);
      return () => {                         // 返回"取消订阅"函数
        listeners[event] = listeners[event].filter(c => c !== cb);
      };
    },
    emit(event, data) {                      // 发布：触发所有订阅者
      (listeners[event] || []).forEach(cb => cb(data));
    }
  };
}
// 关键：挂在 window 上，且用 ||= 保证全局只创建一次
export const bus = (window.__shopBus ||= createBus());
```

`window.__shopBus ||= createBus()` 这行很巧妙：如果 `window.__shopBus` 已存在（别的微前端先加载时建过了），就直接用；不存在才新建。这样 5 个微前端拿到的是**同一个 bus 实例**。

### 事件约定

| 事件名 | 谁发 | 谁收 | 数据 |
|--------|------|------|------|
| `cart:changed` | products / cart / order | shell Header | 购物车总件数 |
| `auth:login` | auth | shell Header | user 对象 |
| `auth:logout` | shell Header（点退出） | 各处 | 无 |

### 实战：购物车角标同步

**发送方**（products 加购后）：
```js
await api('/cart', { method:'POST', body:{productId, quantity:1} });
const cart = await api('/cart');
const count = cart.items.reduce((s, i) => s + i.quantity, 0);
bus.emit(EVENTS.CART_CHANGED, count);   // 广播新数量
```

**接收方**（shell Header）：
```js
useEffect(() => {
  const off = bus.on(EVENTS.CART_CHANGED, (count) => setCartCount(count));
  return off;  // 组件卸载时取消订阅，防止内存泄漏
}, []);
```

> 为什么用事件总线而不是直接调函数？因为 products 编译时根本不知道 shell 的 Header 组件存在——它们是两个独立应用。事件总线让它们**松耦合**：只约定事件名，互不依赖。

## 机制 2：共享依赖（shared）= 共享上下文

`shared: ['react-router-dom']` 让所有微前端用**同一个 react-router 实例**。这带来一个重要能力：

**微前端可以直接用 Shell 的 `BrowserRouter`。**

Shell 在最外层包了 `<BrowserRouter>`，建立了"路由上下文"。这个上下文存在 react-router 的模块内部变量里。因为 react-router-dom 是共享的（同一份实例），所以子微前端调 `useNavigate()` / `useLocation()` / `Link` 时，读到的就是 Shell 那个上下文。

这就是为什么每个微前端的 `App.jsx` 里**不写自己的 `<BrowserRouter>`**，却能直接 `navigate('/cart')`、`<Link to="/products">` 跳转——它们跳的就是 Shell 路由表里的路由。

```jsx
// products/src/App.jsx（简化）
import { useLocation, useNavigate } from 'react-router-dom';

export default function App() {
  const { pathname } = useLocation();     // 读 Shell 当前 URL
  const navigate = useNavigate();          // 用 Shell 的跳转
  // ...
}
```

> 如果每个微前端各自包 `<BrowserRouter>`，就会出现"路由器套路由器"，URL 变了但页面不切。所以记住：**只有 Host 包 BrowserRouter，Remote 直接用 hooks。**

## 机制 3：localStorage = 共享登录态

登录态（token + 用户信息）需要所有微前端都能读到。最简单可靠的方式：存 `localStorage`。

> 代码：`shared/api.js`

```js
export function setAuth(token, user) {
  localStorage.setItem('token', token);
  localStorage.setItem('user', JSON.stringify(user));
}
export function getToken() {
  return localStorage.getItem('token');
}
export function getUser() {
  return JSON.parse(localStorage.getItem('user') || 'null');
}
```

`localStorage` 是浏览器全局存储，同源（同协议+域名+端口）下所有页面、所有 JS 共享。5 个微前端虽然逻辑独立，但都跑在 5173 这个源下（被 Shell 加载），所以读到的 `localStorage` 是同一个。

### 登录态同步流程

```
auth 微前端登录成功
  ├─ setAuth(token, user)          → 写 localStorage（所有微前端可见）
  ├─ bus.emit(EVENTS.LOGIN, user)  → 通知 Shell Header 立即更新 UI
  └─ navigate('/')                  → 跳首页
```

`bus.emit` 是为了"即时"——Header 收到事件马上改 UI；而 `localStorage` 是"持久"——刷新页面后，Header 在 `useEffect` 里读 `getUser()` 恢复登录态。

### 退出登录

Shell Header 的退出按钮：
```js
function logout() {
  bus.emit(EVENTS.LOGOUT);   // 通知各处
  navigate('/');
}
```
Header 自己监听 `LOGOUT` 事件，清空 localStorage 和 user 状态。

## 机制 4：URL = 全局状态

路由 URL 本身就是最朴素的全局状态。例如：

- `/products/3` → 商品微前端从 URL 读 id=3，渲染详情
- `/orders/5` → 订单微前端读 id=5，渲染订单详情

```jsx
// products/src/App.jsx
const match = pathname.match(/^\/products\/(\d+)$/);
if (match) return <ProductDetail id={match[1]} />;  // 从 URL 拿 id
```

好处：可分享、可刷新、可后退。微前端之间跳转只需 `navigate('/路径')`，不用传参。

## 封装：shared/api.js 统一请求

所有微前端调后端都走 `shared/api.js` 的 `api()`，它自动：
- 拼后端地址 `http://localhost:3001/api`
- 从 localStorage 读 token 加 `Authorization` 头
- `JSON.stringify` 请求体
- 解析响应、处理错误

```js
export async function api(path, options = {}) {
  const token = getToken();
  const res = await fetch(BASE + path, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
      ...(options.headers || {})
    },
    body: options.body ? JSON.stringify(options.body) : undefined
  });
  if (!res.ok) {
    const err = await res.json().catch(() => ({}));
    throw new Error(err.error || '请求失败');
  }
  return res.json();
}
```

调用极简：`await api('/cart')`、`await api('/orders', { method:'POST', body:{address, phone} })`。

## 四个机制对比

| 机制 | 解决什么 | 谁用 |
|------|---------|------|
| 事件总线 bus | 实时通知（角标、登录态变化） | products/cart/order/auth ↔ shell |
| 共享依赖 shared | 复用 react、复用路由上下文 | 全部 |
| localStorage | 持久共享登录态 | auth 写，全部读 |
| URL | 资源定位、参数传递 | 全部 |

## 通信时序示例：加购到角标更新

```
用户点"加入购物车"
  │
  ▼
products/ProductList.jsx
  api('/cart', {POST})  ──→ server 写库
  api('/cart')          ──→ server 返回 {items, total}
  bus.emit('cart:changed', count)
  │
  ▼
window.__shopBus 触发所有 'cart:changed' 订阅者
  │
  ▼
shell/components/Header.jsx 的监听回调
  setCartCount(count)  → 角标数字更新 ✓
```
