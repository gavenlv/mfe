# 08 · 进阶：嵌套联邦与多实例

前 7 章我们打下了基础：一个主应用（shell）通过 Module Federation 加载 4 个子应用（products/cart/auth/order），每个子应用在页面上只出现一次。本章讲两个**进阶能力**：

1. **嵌套联邦**：子应用再加载子应用。比如 workspace 这个微前端，本身被 shell 加载，它内部又去加载 products 和 cart。
2. **多实例**：主应用同时加载同一个子应用的多份实例，每份独立运行、状态互不干扰。

这两个能力都建立在前 7 章的基础上，**建议先读 [03 · Module Federation 原理详解](./03-Module-Federation原理详解.md)**，至少搞懂 `exposes / remotes / shared` 三个配置，再回来读本篇。

---

## 一、能力 1：嵌套联邦（Nested Federation）

### 1.1 什么是嵌套联邦

回顾一下普通联邦的结构——它只有两层：

```
shell (host)  ──加载──►  products / cart / auth / order (remote)
```

shell 是宿主（host），它去引用别人；4 个子应用是被加载方（remote）。关系很简单，"我加载你"。

**嵌套联邦是三层**：

```
shell (host)
  └──加载──►  workspace (既是 remote 又是 host)
                ├──加载──►  products (remote)
                └──加载──►  cart (remote)
```

注意 workspace 的位置：相对 shell，它是被加载的 remote；相对 products/cart，它又是加载别人的 host。一个微前端同时扮演两个角色。

**类比"俄罗斯套娃"**：你打开最大的娃娃（shell），里面装着中等娃娃（workspace）；再打开中等娃娃，里面又装着小娃娃（products/cart）。一个微前端里嵌套了另一个微前端。

**应用场景**：假设你要做一个"工作台"页面，把"商品列表"和"购物车"并排放在一页里给运营人员用。这个工作台本身是个独立微前端（可以独立部署、独立开发），但它内部又组合了商品和购物车两个已有的微前端——这就是嵌套联邦的典型用法。本项目就提供了这样一个 `/workspace` 页面。

### 1.2 为什么可行：shared 是全局的

你可能会担心：products 里的 React hooks 调用的是哪一份 React？workspace 里的？shell 里的？会不会出现"多份 React 导致 hooks 报错"？

答案是：**整条链路只有一份 React 实例**。原因在于 Module Federation 的**共享作用域（shared scope）是全局的**。

加载过程是这样的：

1. shell 启动时，加载 `react / react-dom / react-router-dom` 并注册到**全局共享作用域**。
2. shell 加载 workspace，workspace 声明这几个依赖为 `shared`，于是**复用 shell 已加载的那一份**，不再下载自己的。
3. workspace 再加载 products/cart，它们也声明 `shared`，于是**继续复用同一份**。

不管中间隔了多少层，shared 依赖都只加载一次。所以 products 里 `useState` 用的 React，和 shell 里用的是同一个对象，hooks 不会报错。这是嵌套联邦能成立的根本前提。

> 这个特性也解释了为什么第 3 章反复强调"react 必须共享"。如果不共享，普通两层联邦都会出问题，更别说三层嵌套了。

**关键限制**：workspace **不能再包一层 BrowserRouter**。

为什么？因为 `react-router-dom` 是 shared 的，整个应用只有一个 BrowserRouter 才是对的（由 shell 在最外层挂载）。如果 workspace 再包一层，就会出现"多个 Router 实例"，URL 状态、浏览历史全乱套。

正确做法是：workspace 里直接用 `useLocation / useNavigate` 这些 hooks，它们会自动接入 shell 的路由上下文（因为复用的是同一份 react-router-dom）。这一点和 products/cart 的处理方式一样，详见第 6 章。

### 1.3 workspace 的配置：同时是 remote 又是 host

下面是 `workspace/vite.config.js` 的关键配置。仔细看，它**同时拥有 `exposes` 和 `remotes`**——这是嵌套联邦的标志。

```js
federation({
  // 本微前端的唯一标识，shell 里用 import('workspaceApp/App') 引用
  name: 'workspaceApp',
  // 构建产出的远程入口文件名
  filename: 'remoteEntry.js',
  // 作为 remote：对外暴露 ./App，供 shell 远程加载
  exposes: { './App': './src/App.jsx' },
  // 作为 host：声明要引用的下游子应用
  remotes: {
    productsApp: 'http://localhost:5174/assets/remoteEntry.js',
    cartApp: 'http://localhost:5175/assets/remoteEntry.js',
  },
  // 共享依赖，整条联邦链路共用一份 react 等
  shared: ['react', 'react-dom', 'react-router-dom'],
})
```

逐项解释：

- `exposes`：告诉 shell "我提供一个叫 `./App` 的模块"。shell 用 `import('workspaceApp/App')` 来加载它。这一项让 workspace 成为 **remote**。
- `remotes`：告诉 workspace "我要用 productsApp 和 cartApp 这两个远程应用"。这一项让 workspace 成为 **host**。
- `shared`：声明共享依赖。注意它**只写一次**，同时服务于"对外提供"和"对内复用"两个方向——既允许 shell 复用 workspace 的依赖，也允许 workspace 复用 products/cart 的依赖（实际上整条链路复用的是 shell 最先加载的那一份）。

> 这是本项目**首个同时拥有 `exposes` + `remotes` 的配置**。前 7 章里，shell 只有 `remotes`，products/cart 只有 `exposes`，职责分明。workspace 打破了这个划分，一身二任。

完整的 `vite.config.js` 还包括 `build.target: 'esnext'`（联邦需要最新 JS 语法）、`server.port: 5178`（独立开发端口，`strictPort: true` 保证端口稳定，否则 shell 加载不到）。这些和其它子应用一致，不再赘述。

### 1.4 workspace 的 App.jsx：嵌套加载

配置好后，workspace 怎么去加载 products/cart？答案和 shell 加载子应用一模一样——用 `React.lazy` 动态 import，再用 `Suspense` 包裹。下面是 `workspace/src/App.jsx` 的核心代码（省略了 ErrorBoundary 等模板代码）：

```jsx
import { Suspense, lazy, Component } from 'react';

// 动态加载下游子应用（Module Federation 远程模块）
const ProductsApp = lazy(() => import('productsApp/App'));
const CartApp = lazy(() => import('cartApp/App'));

export default function App() {
  return (
    <div>
      <h1 className="page-title">🧩 工作台（嵌套联邦演示）</h1>

      {/* 左右两栏布局：左商品、右购物车 */}
      <div style={{ display: 'grid', gridTemplateColumns: '2fr 1fr', gap: 16, alignItems: 'start' }}>
        {/* 左栏：嵌入商品列表（限定数码分类） */}
        <div className="card">
          <h2>数码商品（来自 products 微前端）</h2>
          <ErrorBoundary name="Products">
            <Suspense fallback={<div className="loading">加载中…</div>}>
              <ProductsApp initialCategory="数码" embedded />
            </Suspense>
          </ErrorBoundary>
        </div>

        {/* 右栏：嵌入迷你购物车 */}
        <div className="card">
          <h2>购物车（来自 cart 微前端）</h2>
          <ErrorBoundary name="Cart">
            <Suspense fallback={<div className="loading">加载中…</div>}>
              <CartApp embedded />
            </Suspense>
          </ErrorBoundary>
        </div>
      </div>
    </div>
  );
}
```

关键点解读：

- `import('productsApp/App')`：写法和 shell 加载子应用完全一样。`productsApp` 是 `vite.config.js` 里 `remotes` 的别名，`/App` 是 products 暴露出来的模块名。区别只是——这次是从 workspace 这个"中间层"发起的加载，形成了三层链路。
- **左栏传 `initialCategory="数码"`**：让这个 products 实例只展示数码分类的商品（而 shell 的 `/products` 路由展示全部）。这就是用 props 区分数据。
- **左右两栏都传 `embedded`**：让子应用进入"嵌入模式"，关闭自身的导航行为（不跳详情、不跳结算），安分地待在工作台里。`embedded` 的具体作用见本章第三节。
- **`Suspense` + `ErrorBoundary`**：远程加载是异步的，首次渲染时模块还没下载完，需要 `Suspense` 显示"加载中…"；加载可能失败（端口没启动、网络问题），需要 `ErrorBoundary` 兜底，否则一个子应用挂掉会让整个工作台白屏。

> 注意 `import('productsApp/App')` 在 workspace 和 shell 里都出现了一次。这不会重复下载——Module Federation 会缓存已加载的远程模块，第二次 import 直接复用。同理 react 等 shared 依赖也只一份。

### 1.5 嵌套联邦小结

- 一个微前端可以**同时是 host 和 remote**：用 `exposes` 对外提供模块，用 `remotes` 引用下游模块。
- **shared 依赖全局共享**，不管联邦隔几层，react 等关键依赖整条链路只有一份，不会出现"多份 React 导致 hooks 报错"。
- **不能再包 BrowserRouter**：要复用宿主的路由上下文，直接用 `useLocation / useNavigate`。
- 适合做"**聚合型**"工作台页面：把多个已有微前端组合进一个新微前端里。

## 二、能力 2：多实例（Multi-Instance）

### 2.1 什么是多实例

普通加载方式下，一个子应用在页面上只出现一次。比如 shell 的 `/products` 路由只渲染一个 `<ProductsApp />`，`/cart` 只渲染一个 `<CartApp />`。

**多实例**：同一个子应用组件在页面上**同时渲染多份**，每份有独立的状态与 props。

本项目的 `/multi` 页面就是演示：4 个 Tab，每个 Tab 是一个子应用实例。其中 3 个是 products 的实例（分别展示数码/电脑/穿戴），第 4 个是 cart 实例。所有实例同时挂载，切 Tab 只切换显示。

**用户的需求场景**往往是这样的：

- 想要多个 Tab，每个 Tab 是一个子应用实例；
- 每个实例只有**数据不一样**（比如 3 个商品列表，分别看不同分类）；
- 或者**子应用和数据都不一样**（比如第 4 个 Tab 切换成购物车，是另一个微前端）。

### 2.2 两种"不同"

多实例里要区分两种"不同"：

**① 同子应用、不同数据**

3 个 products 实例，分别传入 `initialCategory="数码" / "电脑" / "穿戴"`。它们调用同一个 `/api/products` 接口，但带上不同的 `category` 参数，展示不同商品。**组件代码完全一样**，只是 props 不同，所以表现不同。

```
ProductsApp 实例1 (initialCategory="数码")  → 展示数码商品
ProductsApp 实例2 (initialCategory="电脑")  → 展示电脑商品
ProductsApp 实例3 (initialCategory="穿戴")  → 展示穿戴商品
```

**② 不同子应用**

第 4 个 Tab 是 `<CartApp />`，与 products 是**不同的微前端**。这是另一种"不同"——连组件都不是同一个。

### 2.3 实现要点：lazy 组件可渲染多次

很多人会疑惑：`React.lazy(() => import('productsApp/App'))` 返回的 `ProductsApp`，能渲染多次吗？多份实例会不会共享状态？

答案是：**`React.lazy` 返回的就是一个普通组件，可以渲染任意多次，每次都是独立实例**。

```jsx
const ProductsApp = lazy(() => import('productsApp/App'));

// 下面三个是三个独立实例，各自维护自己的 useState，互不影响
<ProductsApp initialCategory="数码" />
<ProductsApp initialCategory="电脑" />
<ProductsApp initialCategory="穿戴" />
```

为什么独立？因为每次渲染 `<ProductsApp />` 都会执行一次组件函数，各自调用 `useState` 拿到各自的状态。`lazy` 只负责"第一次访问时去下载模块代码"，下载完之后它就是个普通组件，React 按正常逻辑管理它的多个实例。**模块代码只下载一次，但实例可以有很多个**。

下面是 `/multi` 页面（`shell/src/pages/MultiInstance.jsx`）的核心代码：

```jsx
import { Suspense, lazy, Component, useState } from 'react';

const ProductsApp = lazy(() => import('productsApp/App'));
const CartApp = lazy(() => import('cartApp/App'));

// Tab 配置：每个 Tab 对应一个子应用实例
const tabs = [
  { key: '数码', type: 'products', initialCategory: '数码', icon: '🎧' },
  { key: '电脑', type: 'products', initialCategory: '电脑', icon: '💻' },
  { key: '穿戴', type: 'products', initialCategory: '穿戴', icon: '⌚' },
  { key: '购物车', type: 'cart', icon: '🛍️' },
];

export default function MultiInstance() {
  const [active, setActive] = useState(0);  // 当前激活的 Tab 索引

  return (
    <div>
      {/* Tab 栏：点击切换 active */}
      <div className="cat-bar">
        {tabs.map((t, i) => (
          <button
            key={t.key}
            className={'cat-chip' + (active === i ? ' active' : '')}
            onClick={() => setActive(i)}
          >
            {t.icon} {t.key}
          </button>
        ))}
      </div>

      {/* 实例容器：所有实例同时挂载，用 display:none 隐藏非激活项 */}
      <div>
        {tabs.map((t, i) => (
          <div key={t.key} style={{ display: active === i ? 'block' : 'none' }}>
            <ErrorBoundary name={t.key}>
              <Suspense fallback={<div className="loading">加载中…</div>}>
                {t.type === 'products'
                  ? <ProductsApp initialCategory={t.initialCategory} embedded />
                  : <CartApp embedded />}
              </Suspense>
            </ErrorBoundary>
          </div>
        ))}
      </div>
    </div>
  );
}
```

逐行解读：

- `tabs` 数组定义 4 个 Tab，每个 Tab 用 `type` 字段区分渲染哪个子应用，用 `initialCategory` 区分数据。
- `active` 是当前激活的 Tab 索引，点 Tab 按钮 `setActive(i)` 切换。
- 实例容器里 `tabs.map(...)` 渲染全部 4 个实例，每个外层 div 用 `display: active === i ? 'block' : 'none'` 控制显示——这就是"全部挂载 + display:none 切换"的关键，详见 2.4。
- 每个 products 实例传 `initialCategory={t.initialCategory}` 和 `embedded`；cart 实例只传 `embedded`。

### 2.4 保持挂载 + display:none 切换（而非条件渲染）

这一节是多实例设计的精髓。先看**错误做法**——条件渲染：

```jsx
// ❌ 错误：只有激活的 Tab 才渲染，其它不渲染
{tabs.map((t, i) => (
  active === i && (
    <ProductsApp initialCategory={t.initialCategory} embedded />
  )
))}
```

这样写的问题：切 Tab 时，旧实例会被**卸载**（unmount），新实例被**挂载**（mount）。卸载意味着状态全部丢失——你在"数码"Tab 里选好的分类筛选、输入的搜索词、滚动到的位置，切走再切回来全没了，每次都要重新加载、重新请求接口。

**正确做法**——全部挂载 + display:none 隐藏非激活项：

```jsx
// ✅ 正确：所有实例都挂载，只切换 display
{tabs.map((t, i) => (
  <div key={t.key} style={{ display: active === i ? 'block' : 'none' }}>
    <ProductsApp initialCategory={t.initialCategory} embedded />
  </div>
))}
```

为什么有效？因为 `display: none` 只是 CSS 层面把元素藏起来，**React 并没有卸载组件**。组件实例一直挂载着，它的 `useState` 状态完整保留。切回来时，UI 立刻按之前的状态显示——这模拟了"多 Tab 浏览器"里每个 Tab 独立保留状态的体验。

**代价**：首次进入 `/multi` 要同时加载全部 4 份远程模块代码。但因为 Module Federation 的 `shared`，react 等关键依赖只加载一份，实际开销主要在各实例的数据请求（4 次 `/api/products` 或 `/api/cart`）。对内网工具型页面来说完全可接受。

> 这是前端常见的"用空间换时间"思路：宁可一次性多加载几个实例，也要换来切换时的流畅体验。如果你只关心当前 Tab、不在乎状态保留，用条件渲染也行；本项目的多实例演示目标是"保留状态"，所以选了 display:none 方案。

### 2.5 多实例小结

- 同一个 `lazy` 组件渲染多次 = 多个**独立实例**（各自独立的 `useState`）。
- 用 **props 区分数据**（`initialCategory`）实现"同子应用不同数据"。
- 用 **`display:none` 切换**（而非条件渲染）保留各实例状态。

## 三、关键设计：embedded 与 initialCategory props

为了让子应用既能当"页面"用（被 shell 路由直接加载），又能当"组件"用（被 workspace 嵌入、在 /multi 里多实例装载），项目给 products 和 cart 加了两个 props。这是嵌套联邦和多实例能跑起来的"配套改造"。

### 3.1 embedded（嵌入模式）

`embedded` 是一个布尔值，true 表示"我正被当组件嵌入，请关闭自身的页面级行为"。

**products 收到 `embedded=true` 时**：

- 隐藏"全部商品"标题（外层容器会提供自己的标题，避免重复）；
- 禁用卡片点击跳详情（在 workspace 工作台里点商品不应该把宿主路由切到 `/products/123`）；
- **仍保留加购功能**（加购是组件级操作，不涉及路由，可以继续用）。

**cart 收到 `embedded=true` 时**：

- 隐藏"购物车"标题；
- 隐藏底部"清空购物车 / 去结算"操作栏。

为什么要隐藏"去结算"？因为去结算会 `navigate('/checkout')`，把宿主路由切走。在工作台或迷你购物车里，这种跳转不合时宜——嵌入场景只展示+增删即可，结算留给用户主动去 `/cart` 页操作。合计信息仍保留，让用户看到当前总价。

**目的**：让子应用变成"纯展示 + 操作"的组件，不干扰外层路由。这是把"页面型"子应用改造为"组件型"子应用的关键开关。

### 3.2 initialCategory（初始分类）

products 的 `ProductList` 组件接受 `initialCategory`，作为 `useState` 的初始值：

```jsx
const [category, setCategory] = useState(initialCategory || 'all');
```

- 不传或传空：默认 `'all'`，展示全部商品。
- 传 `'数码' / '电脑' / '穿戴'`：初始就选中该分类，列表只展示该分类商品。

多个实例传不同值，就能在同一页面上展示不同数据——这就是 2.2 节"同子应用不同数据"的实现机制。注意它是**初始值**，用户在 UI 上仍可以切换分类，各实例的切换互不影响（因为 state 独立）。

### 3.3 products App.jsx 的 embedded 分发

products 的根组件 `App.jsx` 根据 `embedded` 决定渲染逻辑：

```jsx
import { useLocation } from 'react-router-dom';
import ProductList from './components/ProductList.jsx';
import ProductDetail from './components/ProductDetail.jsx';

export default function ProductsApp({ initialCategory, embedded } = {}) {
  const location = useLocation();  // Hooks 必须无条件调用

  // 嵌入模式：始终渲染 ProductList，不做 URL 路由分发
  if (embedded) {
    return <ProductList initialCategory={initialCategory} embedded />;
  }

  // 常规模式：根据 URL 路径分发
  const match = location.pathname.match(/^\/products\/(\d+)$/);
  if (match) return <ProductDetail id={match[1]} />;
  return <ProductList />;
}
```

逻辑解读：

- **嵌入模式**（`embedded=true`）：不管当前 URL 是什么，始终渲染 `ProductList`，并把 `embedded` 和 `initialCategory` 透传下去。这样被 workspace 嵌入或在 /multi 多实例页里时，不会因为宿主 URL 变化而切到详情页。
- **常规模式**：用正则匹配 `/products/数字` 的 URL，命中就渲染详情页，否则渲染列表。这是被 shell 的 `/products/*` 路由加载时的行为。

**强调一条 React Hooks 规则**：`useLocation()` 必须在所有 `return` 之前**无条件调用**。为什么？因为 React 要求 hooks 的调用顺序在每次渲染中保持一致——如果先 `if (embedded) return ...` 再 `useLocation()`，嵌入模式下 `useLocation` 就不会执行， hooks 顺序变了，React 会报错。所以即便嵌入模式用不到 `location`，也要先调用它，再走分支。

> cart 的 `App.jsx` 处理 `embedded` 的方式更直接——在 JSX 里用 `{!embedded && <h1>购物车</h1>}` 和 `{!embedded && <操作栏/>}` 控制显隐，逻辑主体不变。详见 `cart/src/App.jsx`。

为什么必须 `build`？因为 `vite-plugin-federation` 的 `remoteEntry.js` 只在 `vite build` 阶段生成，dev 模式不产出。详见 [03 章"为什么必须 build 才能跑"](./03-Module-Federation原理详解.md)。workspace 既要被 shell 加载（需要自己的 `remoteEntry.js`），又要加载 products/cart（需要它们的 `remoteEntry.js`），所以 6 个前端都要构建。

### 4.3 验证页面

启动后访问主应用 http://localhost:5173 ，验证两个进阶能力：

- **嵌套联邦**：访问 http://localhost:5173/workspace ，看到"工作台"页面——左栏数码商品列表（来自 products），右栏迷你购物车（来自 cart）。这就是 `shell → workspace → products/cart` 三层联邦的产物。
- **多实例**：访问 http://localhost:5173/multi ，看到 4 个 Tab。前 3 个 Tab 是 products 的不同分类实例（同子应用不同数据），第 4 个是 cart 实例（不同子应用）。在各 Tab 里切换分类、输入搜索词，再切到别的 Tab 再切回来，状态都还在——这就是 display:none 保留状态的效果。

顶部导航栏也有"工作台"和"多实例"两个入口，对应 `/workspace` 和 `/multi` 两条路由（在 `shell/src/App.jsx` 里配置）。

## 五、小结表格

| 能力 | 关键机制 | 涉及文件 |
|------|---------|---------|
| 嵌套联邦 | 一个微前端同时配置 `exposes` + `remotes`，shared 依赖全局共享 | `workspace/` 整个目录 |
| 多实例 | `lazy` 组件渲染多次 + `display:none` 切换保留状态 | `shell/src/pages/MultiInstance.jsx` |
| 嵌入模式 | `embedded` / `initialCategory` props 让子应用可当组件用 | `products/src/App.jsx`、`products/src/components/ProductList.jsx`、`cart/src/App.jsx` |

**一句话回顾**：

- 嵌套联邦 = 一个微前端既是 remote 又是 host，靠 shared 全局共享保证只一份 React；
- 多实例 = `lazy` 组件渲染多次得到独立实例，靠 `display:none` 保留各实例状态；
- 嵌入模式 = 用 `embedded` props 把"页面型"子应用改造成"组件型"，用 `initialCategory` 区分多实例数据。
