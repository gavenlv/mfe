## 三个关键协作机制

### 1. Module Federation：前端的"拼装"

Shell 在 `vite.config.js` 里声明了 4 个 remote 的地址：

```js
remotes: {
  productsApp: 'http://localhost:5174/assets/remoteEntry.js',
  cartApp:     'http://localhost:5175/assets/remoteEntry.js',
  // ...
}
```

当用户访问 `/products`，Shell 执行 `import('productsApp/App')`，浏览器就会去 5174 下载 `remoteEntry.js`，再按需下载商品模块的代码，最后渲染出来。**这一切发生在浏览器里，用户感知不到。**

### 2. 共享依赖（shared）：只加载一份 React

5 个前端都依赖 react。如果不共享，浏览器会加载 5 份 react，不仅慢，还会因为"两份 react 实例"导致状态、路由上下文断裂。

通过声明 `shared: ['react', 'react-dom', 'react-router-dom']`，运行时**只有 Shell 最先加载的那一份 react 被使用**，所有微前端共用它。这也是为什么微前端能直接用 Shell 的 `BrowserRouter`（同一个 react-router 实例）。

### 3. 事件总线（bus）：微前端之间的"对讲机"

微前端之间默认互相看不到对方状态。比如：你在商品页加了购物车，Shell 顶部的购物车角标怎么知道要 +1？

解法是 `shared/bus.js` —— 一个挂在 `window.__shopBus` 上的全局事件总线（发布订阅模式）：

- 商品微前端加购后：`bus.emit('cart:changed', 新数量)`
- Shell 头部监听：`bus.on('cart:changed', count => 更新角标)`

同理，登录/登出用 `auth:login` / `auth:logout` 事件广播。

## 一次"下单"的完整数据流

这是理解整个项目最重要的一段。跟着走一遍：

```
[1] 用户在商品页点"加入购物车"
    products/ProductList.jsx
    ├─ 检查 getToken()：没登录 → 跳 /auth/login
    └─ 已登录 → api('/cart', {POST, body:{productId, quantity:1}})
                    │
[2] 请求到达后端 server:3001
    routes/cart.js POST /
    ├─ requireAuth 中间件：验证 JWT，得到 req.user.id
    └─ 写入 SQLite cart_items 表（已存在则数量累加）
                    │
[3] 前端拿到响应，刷新购物车计数
    products 再 GET /api/cart → 算出总件数
    bus.emit('cart:changed', 件数)
    └─ Shell Header 收到事件 → 角标数字 +1  ✓
                    │
[4] 用户去 /cart 页，点"去结算" → 跳 /checkout
    order/Checkout.jsx
    ├─ GET /api/cart 显示订单摘要
    └─ 用户填地址、电话，点"提交订单"
       api('/orders', {POST, body:{address, phone}})
                    │
[5] 后端 routes/orders.js POST /
    用【事务】原子地完成 4 件事：
    ├─ INSERT orders（主订单）
    ├─ INSERT order_items（每个商品一行，含下单时的价格快照）
    ├─ UPDATE products SET stock = stock - 数量（扣库存）
    └─ DELETE cart_items（清空该用户购物车）
                    │
[6] 前端收到 {orderId}
    bus.emit('cart:changed', 0)  → Shell 角标清零 ✓
    navigate('/orders/' + orderId) → 跳订单详情
```

## 为什么后端用 SQLite 而不是 MySQL

- **零配置**：SQLite 是一个文件（`shop.db`），不用装数据库服务，`npm run seed` 就建好
- **够用**：Demo 级数据量，单机，完全够
- **同步 API**：`better-sqlite3` 是同步的，代码好读，不用到处 await 数据库
- **学习友好**：表结构、SQL 语法和 MySQL 几乎一样，学到的能迁移

## 为什么前端要 build 才能跑

`@originjs/vite-plugin-federation` 的 `remoteEntry.js`（微前端的"入口清单"）**只在 `vite build` 时生成**。dev 模式下 Vite 不会产出这个文件，所以 Host 加载不到 Remote。

因此本项目用 `npm run build`（构建）+ `npm start`（`vite preview` 提供构建产物）的方式运行。详见 [03 · Module Federation 原理](./03-Module-Federation原理详解.md)。
